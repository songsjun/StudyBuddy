# 异步协作约定

适用于远程、跨时区、非实时的小团队。

## @mention 必须响应（核心规则）

**任何地方被 `@你的用户名`，无论是 Discussion、Issue、PR、Wiki 还是 commit comment，都必须响应。**

- 响应时限：**48 小时内**（阻塞类 24 小时内）
- 最低响应：哪怕暂时没有答案，也要回复"已收到，X 日前给出意见"
- 不接受无声忽略：没有回复视同没有看到，发起人可以继续 @ 直到得到响应

`@mention` 是团队的异步呼叫机制，是协作能正常推进的基础保障。

## 响应时间约定

| 事项 | 期望响应时间 |
|------|------------|
| 被 @mention（任意位置） | 48 小时内，阻塞类 24 小时内 |
| PR Review 请求 | 48 小时内 |
| 紧急 Bug（标注 `urgent`） | 尽快 |

不要等待即时回复。遇到阻塞，在 Issue 中说明阻塞原因，先做其他事。

## 文字优先原则

- 所有决策和讨论记录在 GitHub，不依赖口头沟通
- 同步会议的结论必须写入对应的 Discussion 或 Issue
- 不要在私信中做影响团队的决策

## 状态透明

- 开始一个 Issue 时：将自己设为 Assignee，添加 `in-progress` label
- 遇到阻塞时：在 Issue 中留言说明原因，添加 `blocked` label
- 暂停一个任务时：在 Issue 中说明进展和下一步，移除 Assignee

## 减少等待的实践

- PR 尽量小，让 Reviewer 容易快速处理
- 设计决策在开发前在 Discussion 中对齐，避免开发完才发现方向错了
- 如果 48 小时没有 Review 响应，可以在 PR 中 @mention 再次提醒
- 同一个时间段内同时推进多个小任务，减少串行等待
