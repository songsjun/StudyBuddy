## 使用方式

在做产品决策前，确保这些问题都有明确答案。
没有答案的问题不等于答案不存在，而是需要先找到答案再做决策。

---

## 1. 问题定义（最重要）

- [ ] **我们在解决谁的问题？** 能描述具体的用户类型吗？
- [ ] **这个问题真实存在吗？** 有用户反馈、数据、还是我们的假设？
- [ ] **不做这个功能，用户会怎样？** 如果答案是"还好"，重新考虑优先级。

## 2. 价值判断

- [ ] **这个功能对用户的价值能用一句话说清楚吗？** 如果说不清楚，功能可能还没想明白。
- [ ] **有多少用户会用到？** 是核心场景还是边缘场景？
- [ ] **这是用户想要的，还是我们认为用户需要的？** 两者经常不一样。

## 3. 范围控制

- [ ] **最小可用版本（MVP）是什么？** 哪些是核心，哪些是"以后再说"？
- [ ] **Out of Scope 明确列出了吗？** 没有明确排除的东西会自动变成范围蔓延。
- [ ] **这个功能会不会让产品变复杂？** 功能增加了，但新用户能快速理解吗？

## 4. 依赖与时机

- [ ] **做这个功能需要什么前提条件？** 基础设施、其他功能、数据？
- [ ] **现在是做这个的正确时机吗？** 还是有更重要的事情先做？
- [ ] **这个功能和我们当前的核心目标是否一致？**

## 5. 成功定义

- [ ] **上线后怎么判断这个功能成功了？** 没有可量化指标的功能很难评估价值。
- [ ] **如果数据表现不好，我们会下线它吗？** 明确 threshold 和应对预案。

---

## 决策记录

做出决策后，在 Discussion 中总结：

```
决策：[Go / No-go / 推迟]

Go 的理由：...
接受的取舍：...
暂不做的部分：...（放入 backlog 或明确丢弃）
下一步：创建 Issue #X，目标 Milestone vX.X
```

## 无法达成共识时的处理

讨论超过 5 天仍无结论时：
1. 发起人总结当前分歧点，列出两种立场各自的核心理由
2. 由 Product Owner 做最终决定（不需要所有人同意）
3. 记录"有保留意见的接受"：`决策：Go（@username 有保留意见，原因记录于此）`
4. "推迟决定"也是有效结论，但必须写明触发重新讨论的条件

## 常见陷阱

- **"用户肯定想要这个"**：没有数据支撑时，这是假设不是事实
- **"做了总比没做好"**：功能有维护成本，烂功能比没有功能更伤害产品
- **"先做出来再说"**：边界不清楚的功能开发完也无法验收
- **"只是小改动"**：小改动累积成大复杂度，每个功能都值得问"值得做吗"
