---
name: checklist/risk-review
description: 识别技术方案和产品决策中的风险敞口
triggers: 评审架构设计、重大技术选型、发布计划时；或在 Discussion 中对方案做风险审查时
---

## 使用方式

对每个风险项评估：**是否存在** + **影响范围** + **缓解措施**。
不是所有风险都需要消除，目的是让团队知晓并主动决策，而不是被动暴露。

---

## 1. 可用性风险

- [ ] **单点故障**：方案中是否存在某个组件故障会导致整个系统不可用？
- [ ] **级联失败**：某个服务变慢或失败时，会不会拖垮其他服务？
- [ ] **外部依赖**：依赖的第三方服务（支付、短信、地图）如果不可用，用户体验会怎样？是否有降级方案？
- [ ] **数据库**：高并发下的锁竞争、慢查询是否会影响可用性？

## 2. 数据风险

- [ ] **数据丢失**：有没有操作会导致用户数据不可恢复地删除？是否有软删除或备份？
- [ ] **数据一致性**：跨多个操作（或微服务）的数据变更，失败时是否会留下不一致状态？
- [ ] **迁移风险**：数据库 migration 是否可回滚？在生产数据量下是否会锁表？

## 3. 范围与依赖风险

- [ ] **隐含假设**：方案是否依赖"用户数据量不会太大"、"并发不会太高"等未验证的假设？
- [ ] **接口契约**：依赖的内部接口如果改变，影响范围是否清楚？
- [ ] **技术债**：方案是否引入了明知未来要返工的临时设计？如果是，是否有记录（见 ADR）？

## 4. 发布与回滚风险

- [ ] **发布风险**：新功能上线是否会影响现有用户的正常使用？
- [ ] **回滚方案**：如果上线后发现严重问题，是否能快速回滚？数据库 migration 是否可逆？
- [ ] **灰度发布**：对于影响大的功能，是否考虑先小范围发布验证？

## 5. 团队与维护风险

- [ ] **知识孤岛**：方案是否只有一个人能理解和维护？
- [ ] **复杂度**：引入的新工具或模式，团队是否有足够的经验？学习成本是否在接受范围内？
- [ ] **可观测性**：新功能出问题时，是否有足够的日志和监控来快速定位？

---

## 输出格式

在 Discussion 或 PR 中写出审查结论：

```
风险审查结论：

高风险（需要在发布前处理）：
- [具体风险描述] → 建议措施：...

中风险（接受并记录）：
- [具体风险描述] → 已知风险，原因：...

无重大风险：...
```

---

## 注意

- 目标是让团队知情并主动决策，不是为了阻止所有有风险的事情
- "已知风险，接受" 是合理的结论，需要记录在 ADR 的"后果"部分
- 不要把每个潜在问题都升级为高风险，这会导致风险审查被忽视
