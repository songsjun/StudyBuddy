triggers: 在合并前验证功能实现符合验收标准时；或审查 PR 中的测试覆盖质量时
---

## 一、对照 PRD 验收

```bash
# 对照 docs/prd/ 下对应的 PRD 文件，逐条检查验收标准
```

- [ ] 每条 Given/When/Then 验收标准是否有对应测试覆盖？
- [ ] 边界值是否有测试（最大值、零、空、特殊字符）？
- [ ] Out of Scope 的内容是否没有被实现进来？

## 二、测试质量审查

**检查现有测试，判断是否有效：**

- [ ] 测试是否在验证行为（输入→输出），而不是在验证实现细节（内部函数被调用了几次）？
- [ ] Mock 是否过度使用？过多 mock 会让测试通过但生产失败。
- [ ] 测试名称是否能描述"测试什么场景"，而不只是"测试某函数"？
- [ ] 是否有只会永远通过的无效测试（没有 assertion，或 assertion 写错了）？

**新增功能必须有的测试类型：**

- [ ] 正常路径（happy path）：功能按预期工作
- [ ] 错误路径：无效输入、权限不足、外部服务失败时的处理
- [ ] 边界情况：空列表、最大值、并发操作（如果适用）

## 三、回归检查

- [ ] 改动是否可能影响现有功能？是否有相关的回归测试？
- [ ] 跑完整测试套件：`npm test`（或项目对应命令），确认无新增失败。
- [ ] 如果改动了共享组件/工具函数，是否检查了所有调用方？

## 四、手动验证（CI 不能替代的部分）

以下场景建议手动验证，CI 难以完全覆盖：

- [ ] UI 交互流程在真实浏览器中是否顺畅（不只是逻辑正确）？
- [ ] 错误提示的措辞是否对用户友好？
- [ ] 在慢网络下（浏览器 DevTools 模拟）功能是否仍可用？
- [ ] 如果有权限控制，是否以不同角色的账号分别验证了？

## 五、测试覆盖率（参考，不是硬指标）

覆盖率不是质量，但极低的覆盖率是信号：

- 核心业务逻辑（计费、权限、数据处理）：目标 > 80%
- UI 组件和工具函数：目标 > 60%
- 配置文件、类型定义：不需要测试

---

## 发现问题时

- 功能性 Bug → 在 PR 创建 `[blocking]` comment，附复现步骤
- 测试遗漏 → `[blocking]`，补充测试后才能合并
- 次要的 UI 细节 → `[suggestion]`，可以在后续 Issue 跟进
- 发现 PRD 中未定义的边界情况 → 回到 Discussion 或直接问 Product Owner
