## 说明

每项只标 ✓（已处理）、✗（存在问题）、N/A（不适用）。
发现 ✗ 时创建 `[blocking]` comment，严重问题阻止合并。

---

## 1. 输入校验

- [ ] 所有来自用户的输入（请求参数、body、header、cookie）在使用前是否校验类型和范围？
- [ ] 文件上传是否校验了文件类型（不只看扩展名，要看 magic bytes）和大小上限？
- [ ] 富文本内容是否经过 HTML sanitize，防止 XSS？
- [ ] 路径拼接是否校验了 `../` 路径穿越？

## 2. 认证

- [ ] 需要登录的接口是否都验证了 session/token 有效性？
- [ ] Token 是否有过期时间？
- [ ] 登录失败是否有频率限制（防暴力破解）？
- [ ] 密码是否使用 bcrypt/argon2 等慢哈希存储？（不是 MD5/SHA1）
- [ ] 敏感操作（改密码、绑定手机）是否要求重新验证身份？

## 3. 授权（最容易被遗漏）

- [ ] 每个资源操作（读、写、删）是否检查了当前用户是否拥有该资源？
- [ ] 仅靠前端隐藏按钮不算授权，后端接口是否有独立的权限验证？
- [ ] 批量操作（批量删除、批量修改）是否逐条校验权限，而不是只校验第一条？
- [ ] 管理员接口是否有独立的角色校验？

## 4. 数据库

- [ ] 所有 SQL 查询是否使用参数化查询或 ORM？没有字符串拼接 SQL？
- [ ] 查询结果是否只返回当前用户有权看到的数据（多租户隔离）？

## 5. 数据暴露

- [ ] API 响应是否只包含必要字段，没有泄露密码 hash、内部 ID、其他用户数据？
- [ ] 错误信息是否过于详细（不要把堆栈信息、SQL 错误返回给客户端）？
- [ ] 日志是否记录了敏感信息（密码明文、完整 token）？

## 6. 依赖与配置

- [ ] 新增的第三方依赖是否有已知 CVE？（`npm audit` / `pip-audit`）
- [ ] 密钥、API key 是否通过环境变量注入，没有硬编码在代码中？
- [ ] 新增的对外接口是否有频率限制（rate limiting）？

---

## 风险等级参考

| 发现的问题 | 处理方式 |
|-----------|---------|
| SQL 注入、认证绕过、任意文件读取 | 立即阻止合并，优先修复 |
| 未授权的越权访问 | 阻止合并 |
| 敏感数据暴露 | 阻止合并 |
| 缺少频率限制 | 根据场景判断，高风险接口阻止合并 |
| 依赖有 CVE | 评估可利用性，中高危阻止合并 |

## 说明

此清单覆盖 OWASP Top 10 的主要场景，面向 Web 应用。
不覆盖：移动端安全、硬件安全、密码学实现细节。
