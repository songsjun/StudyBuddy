triggers: 作为 Reviewer 审查 PR 时
---

## 使用方式

逐项过一遍。发现问题时，在 PR 对应行留 comment，标注级别：
- `[blocking]` 必须修改
- `[suggestion]` 建议，作者自行决定
- `[question]` 提问，不要求改动

---

## 1. 正确性

- [ ] 核心逻辑是否和 PR 描述中声称的行为一致？
- [ ] 边界值是否处理：空值、零值、最大值、空列表、空字符串
- [ ] 异步操作是否有竞态条件（race condition）风险？
- [ ] 错误路径是否有处理（不只是 happy path）？
- [ ] 如果依赖外部服务，外部服务不可用时是否有降级？

## 2. 安全性（快速扫描，详细见 checklist/security-review）

- [ ] 用户输入在使用前是否经过校验？
- [ ] 数据库查询是否使用参数化查询，没有拼接 SQL？
- [ ] 返回给前端的数据是否包含不该暴露的字段（密码 hash、内部 ID 等）？
- [ ] 权限检查：操作前是否验证了当前用户有权限执行？

## 3. 可维护性

- [ ] 函数/方法是否只做一件事？超过 40 行值得质疑。
- [ ] 变量名和函数名是否能自解释（不需要注释才能理解）？
- [ ] 有无魔法数字？应替换为有名字的常量。
- [ ] 重复代码超过 3 次？考虑提取函数。
- [ ] 新增的复杂逻辑是否有注释解释"为什么"（不是"做了什么"）？

## 4. 测试

- [ ] 新功能是否有对应测试？
- [ ] 测试是否覆盖了边界情况，不只是最简单的 case？
- [ ] 修复 Bug 的 PR 是否包含能复现该 Bug 的测试？

## 5. PR 范围

- [ ] PR 是否只做了它描述的事？没有混入不相关的改动？
- [ ] 有无遗留的调试代码（`console.log`、TODO 注释、注释掉的代码块）？

---

## 不需要在 Review 中指出的

- 代码风格（这是 linter 的工作）
- 个人命名偏好（camelCase vs snake_case，linter 已约束）
- 实现方式的主观偏好（"我会这样写"），除非有明确的性能或可维护性理由

## Review 完成标准

所有 `[blocking]` 问题已列出。无 blocking 问题时 Approve，有时 Request Changes。
不要因为还有 `[suggestion]` 而不 Approve，建议由作者自行决定是否采纳。
