name: github/pr-create
description: 按项目规范创建 Pull Request
triggers: 功能分支开发完成，本地测试通过，准备提交 Review 时
---

## 前置检查

```bash
# 确认在正确的分支上
git branch --show-current

# 确认本地领先于 main，没有落后
git fetch origin
git log origin/main..HEAD --oneline

# 确认 CI 本地可过（至少跑一次测试）
npm test  # 或项目对应的测试命令
```

## 推送分支

```bash
git push -u origin HEAD
```

## 创建 PR

```bash
gh pr create \
  --title "[feat] 添加课程收藏功能" \
  --body "$(cat <<'EOF'
## 做了什么
支持用户收藏课程，最多 100 门，超出时提示。

## 为什么这样做
产品需求 #23，用户调研中高频请求的功能。
Closes #23

## 如何测试
1. 登录后进入课程页，点击收藏按钮
2. 验证收藏列表正确显示
3. CI 已覆盖单元测试
EOF
)"
```

**标题格式**：`[type] 简短描述`，type 取值：`feat` / `fix` / `docs` / `refactor` / `test` / `chore`

## 指定 Reviewer

```bash
gh pr edit --add-reviewer username1,username2
```

如果是 AI Agent 生成的代码，在 PR body 末尾注明：
```
---
Generated by AI Agent (Claude). Please review carefully.
```

## PR 大小原则

- 单个 PR 理想在 400 行以内（不含测试和生成代码）
- 超出时考虑拆分：先合并基础设施 PR，再合并功能 PR
- 大 PR 是 Review 质量下降的最常见原因

## 注意

- 不要在一个 PR 里混入格式清理、重构和功能变更，难以 Review
- 创建后检查 CI 是否触发：`gh pr checks`
- 48 小时无人 Review，@mention 相关成员
