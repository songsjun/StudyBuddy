## 什么时候写 ADR

写 ADR 的判断标准（满足任意一条）：
- 有两个以上可行方案，做出了选择
- 决策会影响超过一个模块
- 未来的开发者可能会疑惑"为什么这样设计"
- 决策有明显的取舍（tradeoff）

**不需要写 ADR**：实现细节、代码风格、明显唯一的技术选择。

## 文件位置与命名

```
docs/adr/001-选择数据库.md
docs/adr/002-认证方案.md
docs/adr/003-前端框架选型.md
```

序号递增，标题用中文描述决策的主题。

## 模板

```markdown
# ADR-{序号}：{决策主题}

- **状态**：已接受 / 提议中 / 已废弃 / 已替代（被 ADR-{序号} 替代）
- **日期**：YYYY-MM-DD
- **决策者**：@username

## 背景

[描述面临的问题或需求，以及为什么需要做这个决策。
2-4 句话，不要写实现细节。]

## 考虑的方案

### 方案 A：{名称}
优点：...
缺点：...

### 方案 B：{名称}
优点：...
缺点：...

## 决策

选择**方案 A**。

[1-3 句话说明选择理由，重点是"为什么选这个而不是其他方案"。]

## 后果

**正面影响**：
- ...

**负面影响 / 风险**：
- ...

**需要做的事**：
- [ ] ...
```

## 写作要点

- **背景** 写问题，不写答案。让读者理解决策时的上下文。
- **方案** 要诚实列出被否决方案的优点，否则读者不信任你的判断。
- **决策** 要明确，不要模糊表述如"综合考虑选择A"，要说清楚关键理由。
- **后果** 必须包含负面影响，没有完美决策，承认代价才可信。

## 状态更新

当决策被推翻或替代时，不要删除旧 ADR，更新状态字段并引用新 ADR：
```
- **状态**：已替代（被 ADR-008 替代，原因：性能瓶颈）
```
