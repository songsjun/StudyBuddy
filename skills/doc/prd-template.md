## PRD 的边界

PRD 回答"做什么"和"为什么做"，不回答"怎么做"（那是技术设计的事）。
PRD 完成的标志：开发者读完知道该实现什么，QA 读完知道怎么验收。

## 文件位置

```
docs/prd/feature-course-favorites.md
docs/prd/feature-recommendation.md
```

## 模板

```markdown
# PRD：{功能名称}

- **状态**：草稿 / 评审中 / 已定稿 / 已废弃
- **作者**：@username
- **日期**：YYYY-MM-DD
- **关联 Discussion**：#{discussion-number}
- **关联 Milestone**：v1.x.0

## 一句话描述

[用一句话描述这个功能为哪类用户解决了什么问题。]

## 背景与动机

[为什么要做这个功能？用户痛点是什么？不做会有什么影响？
2-5 句话，面向团队成员，不面向用户。]

## 用户故事

- 作为 **{用户类型}**，我希望 **{做某件事}**，以便 **{获得某种价值}**。
- 作为 **{用户类型}**，我希望 **{做某件事}**，以便 **{获得某种价值}**。

## 功能范围

### 包含（In Scope）
- ...
- ...

### 不包含（Out of Scope）
- ...（明确排除，防止范围蔓延）

## 验收标准

每条标准可被明确测试，格式：Given / When / Then

- **Given** 用户已登录，**When** 点击课程页的收藏按钮，**Then** 课程出现在收藏列表中
- **Given** 收藏已满 100 门，**When** 再次收藏，**Then** 提示"收藏已达上限"
- ...

## 边界情况与限制

- 最多收藏 {N} 门课程
- {其他数量、权限、状态限制}

## 成功指标（可选）

[如果有可量化的目标，写在这里。没有的话删掉本节。]
- 功能上线 2 周内收藏使用率达到 X%
```

## "已定稿"的标准

PRD 可以进入开发的判断：
1. 验收标准中没有"可能"、"也许"等模糊表述
2. Out of Scope 明确列出了团队讨论过但排除的内容
3. 至少一名开发者读完并确认理解（在 Discussion 中回复）
4. 关联的 Issues 已创建并加入 Milestone
