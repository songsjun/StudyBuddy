## 受众与原则

Release Notes 的读者是**用户和团队成员**，不是开发者。
- 写用户感知到的变化，不写实现细节
- "修复了课程页面加载慢的问题" > "优化了 API 响应缓存逻辑"
- 破坏性变更必须显眼，并提供迁移方式

## 生成草稿

从 git log 提取提交，生成初稿：

```bash
# 获取上个版本以来的提交
PREV_TAG=$(git tag --sort=-version:refname | head -2 | tail -1)
git log ${PREV_TAG}..HEAD --oneline --no-merges
```

按 commit type 分类整理，过滤掉 `chore`、`refactor`、`test` 等用户无感知的变更。

## 模板

```markdown
## v{版本号} ({日期})

{一句话概括本次版本的主要主题，例如：本版本聚焦于学习体验优化。}

### 新功能
- **课程收藏**：支持收藏最多 100 门课程，在个人中心快速访问 (#23)
- **{功能名}**：{用户视角的一句话描述} (#{issue})

### 改进
- 课程搜索响应速度提升约 40% (#31)
- {其他改进}

### 修复
- 修复在 Safari 浏览器下视频播放中断的问题 (#28)
- {其他修复}

### 破坏性变更（Breaking Changes）
{如有，必须列出，并说明如何迁移。没有则删除本节。}

- **API 变更**：`/api/v1/courses` 返回格式调整，详见 [迁移指南](链接)
```

## 写作检查

发布前确认：
- [ ] 破坏性变更单独列出且显眼
- [ ] 每条内容关联了 Issue 编号（方便溯源）
- [ ] 没有内部术语（如"重构了服务层"这类用户不理解的描述）
- [ ] "新功能"和"改进"分开（新功能是没有的变成有，改进是有的变得更好）

## 发布到两个地方

1. **GitHub Release**（创建 Tag 时一起发布）
2. **Wiki → Release-History.md**（追加到页面顶部，供用户长期查阅）
